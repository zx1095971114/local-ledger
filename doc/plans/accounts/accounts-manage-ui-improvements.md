## 2. 总体结论

页面**信息结构正确**（汇总 → 整体分布 → 分类折叠 → 表格操作），但存在若干主要问题：

1. **纵向与横向留白过多**，关键数字与图表在视区内「占比」偏低，桌面端信息密度未充分利用。  
2. **说明性文案过长且位置靠前**，与图表区右侧说明叠加后，**视觉层级偏弱**，用户第一眼难以区分「可操作区」与「解释性文字」。  
3. **分类内饼图采用固定策略**，在「仅 1～2 个正余额账户」时**信息增益接近零**，却占用与多账户类别相近的宽度，造成**布局浪费与认知噪音**。  
4. **类别较多时，鼠标滚轮无法上下滚动页面**，下方折叠分组不可达（问题分析见 **§3.9**，状态见 **§1 总览表**）。  
5. **分类内饼图列宽过窄、表格列过宽**，饼图可读性差，与「分类占比」诉求不匹配（问题分析见 **§3.10**，状态见 **§1 总览表**）。

以下分项说明，并给出可落地的改进项（含优先级）；各节末尾 **实现状态** 与 **§1 总览表**一致。

## 3. 问题分项

### 3.1 页头与说明文案

**现象**

- 标题下整段灰色说明（负债、饼图分母、净资产与总资产差异）常驻展示，**占据首屏垂直空间**。  
- 与下方「整体分布」卡片内的说明（净资产公式）**语义部分重复**，用户需要读两遍才能建立完整心智模型。

**影响**

- 首屏有效内容（汇总数字、整体饼图）下移。  
- 重要操作（搜索、新增）与长文案挤在同一扫视路径上，**主次不清**。

**改进方向**

- 将页头副标题**缩短为一行**或改为「问号」**Tooltip / 抽屉**（例如：「口径说明」链接）。  
- 卡片内右侧说明与页头说明**合并为一份**（只保留一处），或右侧改为**要点列表 + 链接**跳转 spec。

**优先级：P1**

**实现状态：✅** 见 §1 表 §2.1。

---

### 3.2 汇总卡片（净资产 / 总资产 / 总负债）

**现象**

- 卡片（或 `summary-cell`）**内边距偏大**，标签字号相对较小，数字与容器比例失调。  
- 三列在宽屏下**横向拉得很开**，中间区域易产生「空」的感受。

**影响**

- 核心 KPI 不够「抓眼」，与下方大块图表区相比**权重反而偏弱**。

**改进方向**

- 收紧 `padding`，或改用 **`el-statistic`** / 紧凑卡片，使**数字字号 ≥ 当前标题级**的视觉权重。  
- 考虑 **单行三列等分 + 竖向分隔线**，或 **横向紧凑条**（非三块独立大灰底），减少「三块大方砖」感。  
- 总资产 / 总负债与整体饼图存在语义关联，可增加**微弱视觉串联**（例如整体卡片顶部重复一行小号「总资产 ¥xxx」与饼图分母一致，或仅在 tooltip 中强调）。

**优先级：P1**

**实现状态：✅** 见 §1 表 §2.2。

---

### 3.3 整体分布卡片：图表与右侧留白

**现象**

- 左侧饼图 + 右侧说明的 **两列布局在 lg 断点下中间空隙明显**；说明文字篇幅短，无法填满列宽。  
- 图例置于饼图下方，**整卡垂直高度由图表+图例主导**，右侧文字区垂直居中感弱，易出现「右半屏空」的观感。

**影响**

- 用户注意力被「空白」打断，误以为页面未加载完全。  
- 整体饼图作为**最重要可视化**，其容器却像「未排满的 PPT」。

**改进方向**

- **窄化右侧说明列**（如 `lg` 下改为 `span=6~8`），或改为**图表下方全宽一行**的脚注（与图例同一视觉块）。  
- 宽屏时采用 **左图右表**：右侧不放长文，改为 **Top 账户简表**（名称、余额、占比），与饼图数据同源，**提高信息密度**。  
- 若保留文字说明：使用 **`el-alert` type="info"** 一行密度 + 展开，或 **`el-collapse` 默认折叠**「口径说明」。

**优先级：P0**（对首屏观感影响最大）

**实现状态：✅** 见 §1 表 §2.3。

---

### 3.4 分类折叠面板：标题与箭头

**现象**

- Element Plus `el-collapse` 默认将**展开箭头置于行尾**，与左侧类别标题距离远，**扫视时需要左右跳读**。  
- 类别标题仅「现金 (1)」，**缺少该类别小计余额**（可选），对比账单类页面习惯略弱。

**影响**

- 多类别时，**定位当前分组**成本略高。

**改进方向**

- 使用 **自定义 title 插槽**：左侧 **箭头 + 类别名 + 账户数**，必要时右侧对齐 **类别余额小计**（正余额合计或净值，需与 spec 一致并 tooltip 说明）。  
- 或评估 **`el-collapse` 换为 `el-card` + 自带 `el-collapse` 内层**，统一边距与层级。

**优先级：P2**

**实现状态：✅** 见 §1 表 §2.4（展开箭头仍在组件行尾，未改 EP 默认位置）。

---

### 3.5 分类内饼图：单账户 / 少账户场景

**现象**

- 每个分组固定 **左图右表**，当该组仅 **1 个正余额账户** 时，饼图为**整圆 100%**，与表格信息**完全重复**。  
- 左列仍占 `md=10 / lg=9` 比例，**横向浪费**，且引导用户「看图」却无新增信息。

**影响**

- 违反「图表应补充表格无法一眼表达的信息」的常见原则；界面显得**为图而图**。

**改进方向（与 spec 兼容）**

- **条件渲染**：当 `pieItems.length <= 1` 时，**不展示饼图**，改为 **一行文字**：「本类别仅一个正余额账户，占比 100%」或 **迷你进度条**；表格全宽。  
- 当 `pieItems.length === 2` 时，可选 **极小号饼图**（高度 ≤140px）或 **堆叠条形对比**，仍占较小列宽。  
- `pieItems.length >= 3` 时维持当前饼图 + 表布局。

**优先级：P0**（直接减少冗余与空白）

**实现状态：⚠️** 见 §1 表 §2.5（两扇区策略与原文「≤140px 或条形」不完全一致）。

---

### 3.6 表格与操作列

**现象**

- 「编辑 / 账单 / 删除」均为 **link 按钮**，字号小且**间距紧**，密集点击区易误触。  
- **整行可点击**跳转账单，与「编辑」等操作**同一手势区域**，部分用户不清楚行点击含义（缺少 hover 提示或光标样式）。

**影响**

- 可发现性与误触风险并存。

**改进方向**

- 操作列改为 **`el-button text` 带图标** 或 **图标按钮 + Tooltip**，并统一 **`gap`**。  
- 表格增加 **`highlight-current-row`** 或行 hover 提示文案（如 `title`）：「单击行查看该账户账单」。  
- 若保留行点击：考虑 **仅「名称」列可点**（链接样式），其余行区域不触发跳转，降低误触。

**优先级：P1**

**实现状态：✅** 见 §1 表 §2.6（未加 `highlight-current-row`，以名称链接 + 文案提示替代）。

---

### 3.7 饼图配色与对比度

**现象**

- ECharts 默认调色在浅色主题下**偏柔和**，图例项多时 **相邻色区分度**一般；引导线标签在窗口缩放时可能**拥挤**。

**影响**

- 长时间扫读时**辨识度**略差（尤其色弱用户）。

**改进方向**

- 在 `AccountPieChart` 中注入 **主题感知 color palette**（与 Element Plus 主色协调的 **高区分度序列**）。  
- 账户数 > 6 时：**图例滚动 + 合并尾部为「其他」**（spec 已允许 V2，可提前做阈值）。  
- 标签策略：`label` **仅显示百分比**或 **仅 tooltip 显示金额**，减少扇区外线段交叉。

**优先级：P2**

**实现状态：⚠️** 见 §1 表 §2.7。

---

### 3.8 响应式与窄屏

**现象**

- 分类区 `el-row` 在 **xs** 下图表在上、表在下，合理；但 **整体分布区** 若仍保留「大高度饼图 + 短文字」，窄屏下**滚动距离偏长**。

**改进方向**

- 小屏下整体饼图高度 **略减**（如 220 → 200），说明 **默认折叠**。  
- 工具栏搜索与排序在 **xs** 下 **纵向满宽堆叠**（当前 `flex-wrap` 已部分支持，可检查 `toolbar-search` 是否 `width:100%`）。

**优先级：P2**

**实现状态：⚠️** 见 §1 表 §2.8。

### 3.9 页面滚动：类别多时无法滚到下方分组

**现象**

- 账户**类别较多、内容总高度超过视口**时，用户用**鼠标滚轮**无法继续向下滚动，**下方的折叠分组看不到**。  
- 部分环境下表现为：滚轮事件被「吃掉」或焦点落在不可滚动容器上，页面主体不跟随滚动。

**可能原因（实现侧需排查）**

- 父级布局链路上存在 **`overflow: hidden`**（例如 `Layout.vue` 中 `.app-main`、`.main-content` 等为撑满高度而限制溢出），子页面虽设 `overflow: auto`，但**实际可滚动层不是当前接收滚轮的那一层**，导致滚轮无效。  
- **`flex` 子项**未正确设置 `min-height: 0` / `overflow`，中间某层把高度锁死为视口高，内层内容被裁切且无法滚动。  
- **典型根因（本项目已遇）**：顶栏下的**内层 `el-container`（侧栏 + 主区）**未参与纵向收缩（缺 **`flex: 1; min-height: 0`**），其高度随内容等于整页高度；**`.main-content` 被撑得与内容同高**，内部**无溢出**，滚动条不出现，超出部分仅被最外层 **`overflow: hidden`** 裁掉。  
- 鼠标悬停在 **ECharts canvas**、**折叠面板**等子区域时，部分浏览器对滚动冒泡行为不一致，加剧「滚不动」感受。

**影响**

- **功能性缺陷**：超出首屏的内容无法访问，与「账户管理」多类别场景直接冲突。

**改进方向**

- 在 **`Layout.vue` → `router-view` 包裹层**与**账户页根节点**上明确约定：**唯一纵向滚动容器**（例如仅 `el-main` 内 `.main-content` 或仅页面根 `overflow-y: auto`），避免多层互相冲突；链路上 **`min-height: 0`** 配齐。  
- 账户页根节点使用 **`height: 100%` + `min-height: 0` + `overflow-y: auto`**（或与布局文档统一的一种写法），并在真机用「多类别 + 长列表」回归验证滚轮。  
- 若 ECharts 区域拦截滚轮：对图表容器设置 **`pointer-events` 策略**或在外层统一滚动（按产品偏好二选一）。

**优先级：P0**（阻塞浏览全部类别，属可用性缺陷）

**实现状态：✅** 见 §1 表 §2.9：含 **`.app-body`** 修复内层容器高度链；未单独做 ECharts 区域 `pointer-events`。

### 3.10 分类内布局：饼图过小、表格抢占版面

**现象**

- 每个类别分组采用 **左饼图、右表格** 的栅格比例（当前宽约 **`md=10` / `lg=9` 图 + `md=14` / `lg=15` 表**），表格列（名称、余额、备注、排序、操作）在宽屏下**横向扩展明显**，左侧饼图区域**相对过窄**。  
- 饼图固定高度约 **220px**，在**窄列宽**下环形半径小，**扇区、引导线、百分比文字**难以辨认，图例也相对局促。

**影响**

- 用户难以快速完成「看分类内资产结构」这一任务，**图表形同虚设**；与 [accounts-manage-spec.md](accounts-manage-spec.md) 中「每类一张饼图」的产品意图不符。

**改进方向**

- **提高图列最小宽度占比**：例如宽屏改为 **图 `lg=12~14`、表 `lg=12~10`**，或 **图列 `min-width: 320px~400px`**，表格列使用 `min-width` + 横向滚动而非无限吃宽度。  
- **上下布局（推荐一种）**：在 **`md` 及以上** 也可采用 **上图下表**：饼图 **全宽**、高度 **260～320px**，表格在下方占满宽，避免「图表抢同一行宽度」。  
- **仅多扇区时大图**：与 §2.5 联动——`pieItems.length <= 1` 不画图；**≥2 且需要饼图**时再用**较大尺寸**展示，避免「小图 + 大表」同时出现。  
- 在 `AccountPieChart` 中根据**容器宽度**调用 `resize`，并考虑 **`radius` 略增大**、标签 **收进 tooltip** 以减轻小图上的文字堆叠。

**优先级：P0**（直接影响分类饼图可读性）

**实现状态：✅** 见 §1 表 §2.10。
